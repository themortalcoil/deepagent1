## Critical Discovery

### The current uncommitted `src/agent.py` patch is BROKEN

The patch imports `WriteFileSchema` from `deepagents.middleware.filesystem`, but **this class does not exist in deepagents v0.4.12** (the installed version). It only appears in v0.5.3+. This means `from src.agent import agent` **will crash at import time** with an `ImportError`.

The two monkey-patches in the uncommitted diff:
1. `FilesystemBackend.write` — works (dict-to-str coercion before calling original)
2. `WriteFileSchema` subclass + global replacement — **broken** (import fails on v0.4.12)

### The correct strategy: Subclass FilesystemBackend, don't monkey-patch

Research confirms:
- `FilesystemBackend` is designed to be subclassed (has `write()` and `awrite()` methods)
- `create_deep_agent()` accepts a `backend=` parameter for custom backends
- No upstream support for dict content exists (even in v0.5.3, `WriteFileSchema.content` is `str` only)
- There are no middleware hooks, config options, or "harness profiles" that handle this

**Option B (Subclass FilesystemBackend)** is the recommended approach:
- Clean, testable, no monkey-patching
- Works with the installed v0.4.12
- Won't break on library upgrades
- The `create_deep_agent(backend=...)` parameter accepts any backend instance

For the frontend, the `client.runs.stream` patch is the correct approach (no built-in stream-mode filtering exists), but the TypeScript types need fixing via strategic `any` casts since JS overloads can't be expressed in a single function assignment.

### Task 1: Fix Python agent — Replace monkey-patches with FilesystemBackend subclass [HIGH]

**Files:** `src/agent.py`

**Current state:** 50 lines of broken monkey-patching (WriteFileSchema import crashes on v0.4.12)

**New approach:**

```python
import json
from deepagents.backends.filesystem import FilesystemBackend
from deepagents.backends.protocol import WriteResult

class RobustFilesystemBackend(FilesystemBackend):
    """FilesystemBackend that auto-serializes dict content to JSON strings.
    
    Some models (e.g., GLM) pass write_file content as a dict instead of str.
    This subclass coerces any non-string content before passing to the parent.
    """

    def write(self, file_path: str, content) -> WriteResult:
        if isinstance(content, dict):
            content = json.dumps(content, indent=2)
        elif not isinstance(content, str):
            content = str(content)
        return super().write(file_path, content)

    async def awrite(self, file_path: str, content) -> WriteResult:
        if isinstance(content, dict):
            content = json.dumps(content, indent=2)
        elif not isinstance(content, str):
            content = str(content)
        return await super().awrite(file_path, content)
```

Then pass it to `create_deep_agent`:

```python
backend = RobustFilesystemBackend(root_dir=str(_PROJECT_ROOT))

agent = create_deep_agent(
    model=_build_model(ORCHESTRATOR_MODEL),
    system_prompt=(...),
    skills=[f"{SKILLS_DIR}/"],
    backend=backend,  # <-- custom backend replaces monkey-patched default
    subagents=[...],
)
```

**Remove entirely:**
- The `WriteFileSchema` import and `PatchedWriteFileSchema` class (lines 74-102)
- The `FilesystemBackend.write` monkey-patch (lines 59-72)
- Unused imports: `from deepagents.middleware.filesystem import WriteFileSchema`, `from pydantic import field_validator`

**Error 2: `useDeepAgent.ts:52` — client.runs.stream type mismatch**

The patched function only accepts `threadId: string` but the original has two overloads (`string | null`). Fix by casting through `any` for the assignment, since we can't express multiple overloads in a single function expression:

```typescript
function createPatchedClient() {
  const client = new Client({ apiUrl: API_URL })
  const origStream = client.runs.stream

  // Cast through `any` — we're intercepting all overloads and delegating
  // to the original, which handles overload dispatch at runtime.
  // eslint-disable-next-line @typescript-eslint/no-explicit-any
  ;(client.runs as any).stream = function(
    threadId: any,
    assistantId: string,
    payload?: any,
  ) {
    if (payload?.streamMode) {
      const modes = Array.isArray(payload.streamMode)
        ? payload.streamMode
        : [payload.streamMode]
      const filtered = modes.filter((m: string) => SUPPORTED_STREAM_MODES.has(m))
      payload = {
        ...payload,
        streamMode: filtered.length > 0 ? filtered : ['values'],
      }
    }
    return origStream(threadId, assistantId, payload)
  }
  return client
}
```

## Risks & Tradeoffs

- **RobustFilesystemBackend**: Subclassing is the intended extension pattern. If deepagents changes the backend interface in v0.5.x, we need to update `awrite()` signature too. The subclass approach means we only maintain the override, not a full replacement.
- **TypeScript `any` casts**: The client.runs.stream patching requires `any` because TypeScript can't express multiple overloads in a single function expression. This is acceptable for a runtime proxy pattern. The alternative (no filtering) would cause 422 errors from langgraph-cli 0.4.x when `useStream` auto-tracks `"tools"` mode.
- **deepagents v0.5.x upgrade**: Deferred. v0.5.3 has breaking API changes (new `permissions` parameter, `_HarnessProfile`, `_ToolExclusionMiddleware`). The current v0.4.12 works fine with our subclass approach. Pin `deepagents>=0.4.8,<0.5.0` in pyproject.toml to prevent accidental breakage.
